# Capital Structures, Inc. — Updated Site Text + Interaction Brief

This document codifies the updated language, structure, and interaction model for the next version of the site. It is written to be directly useful inside Claude Code as a build brief and copy deck.

## 1. Core Positioning

### Primary line

**Capital Structures, Inc. studies and publishes the legal and administrative architectures through which risk is transferred, isolated, priced, guaranteed, and disclosed.**

### Expanded positioning

Capital Structures, Inc. is not about finance in the abstract. It is about how exposure is arranged in space: across entities, guarantees, affiliates, triggers, notes, series, vehicles, covenants, and disclosures. It studies structures, infrastructures, and the legal architectures that permit risk to be moved, contained, repriced, deferred, documented, and viewed.

The project treats diagrams, documentation, and entity maps not as support materials but as the work itself. The structure is the subject. The diagram is the product.

### Short version for homepage / metadata

Capital Structures, Inc. publishes diagrams, documents, and structural studies of how risk is transferred, isolated, priced, guaranteed, and disclosed.

### Internal thesis

The project is strongest when it does **not** say that structures are fake. It says that legitimacy, beauty, containment, opacity, and danger are often produced by the same mechanisms.

## 2. Core Doctrine

These are the governing propositions of the project. They should shape the homepage, doctrine page, glossary, structure pages, and launch video.

### Doctrine 01 — Risk does not disappear

Risk is not erased. It is transferred, isolated, repriced, deferred, ring-fenced, guaranteed, made non-recourse, or disclosed.

**Working line:**

**Securitization is presented not as concealment, but as a legitimate transfer, isolation, and pricing of risk.**

### Doctrine 02 — Exposure is conserved

**Exposure Conservation** is one of the signature concepts of the site.

**Definition:** The proposition that risk displaced from one layer of a structure must reappear elsewhere rather than disappear.

**Capital Structures usage:** A doctrine of structural thermodynamics.

This concept should appear in the glossary, doctrine page, launch video, and possibly as a recurring line inside diagram captions or overlays.

### Doctrine 03 — Bankruptcy is architectural

Bankruptcy remoteness, non-recourse, entity separation, and ring-fencing should be treated not merely as legal mechanics but as spatial operations.

The project is about how exposure is arranged in space.

### Doctrine 04 — Disclosure is not transparency

Disclosure is the conversion of danger into document.

A structure may be fully documented and still remain difficult to perceive in the aggregate.

### Doctrine 05 — Consolidation reveals what separation conceals

Each entity may present a clean and authorized view of itself. The aggregate structure may reveal persistent dependencies, guarantees, residual exposures, and live trigger paths.

This is one of the central tensions of the project.

### Doctrine 06 — The diagram is an argument

A diagram should not merely illustrate a structure. It should test its claims: isolation, recourse limits, guarantees, trigger conditions, failure paths, and residual exposure.

## 3. Naming System for Views and States

The project language should be tightened around the user’s preferred terms: **consolidated** and **unconsolidated**.

### Recommended primary toggle labels

**UNCONSOLIDATED / CONSOLIDATED**

These are stronger and more project-specific than Legal View / Aggregate View.

### Helper copy beneath or beside the toggle

- **Unconsolidated:** Legal/entity view
- **Consolidated:** Aggregate/exposure view

### Meaning of each mode

#### Unconsolidated
The clean entity map. Formal separations are respected. Series, vehicles, affiliates, guarantees, and ownership are shown as discrete units. This is what a structure is allowed to say about itself.

#### Consolidated
The aggregate exposure map. Guarantees, dependencies, recourse edges, reputational links, liquidity dependencies, and contingent obligations are overlaid across the structure. This is what the same structure looks like when viewed in the aggregate.

### Recommended state toggle labels

**BASE CASE / TRIGGER STATE**

Alternative labels if desired:
- Authorized / Triggered
- At Rest / Activated
- Filed / Triggered

But **Base Case / Trigger State** is the clearest and strongest pairing.

### Meaning of each state

#### Base Case
The calm, authorized, filing-ready state. The structure is operative. Obligations are current. Triggers have not fired. Guarantees remain contingent. Liquidity remains available.

#### Trigger State
A contingent state activated by downgrade, valuation shortfall, covenant breach, liquidity freeze, guarantee event, margin demand, cross-default, impairment, or other contractual condition. The structure is no longer only diagrammed. It is in motion.

## 6. Doctrine / Method Page Copy

### Page intro

This page records the governing propositions by which Capital Structures, Inc. studies and publishes structures. These propositions are not exhaustive. They may be amended, expanded, superseded, restated, or triggered by future formations, guarantees, affiliate arrangements, disclosures, impairments, or consolidations.

### Section: Risk does not disappear

A structure does not abolish exposure. It determines where exposure appears, under what name, at what cost, with what recourse, and at what distance from the originating entity.

### Section: Exposure is conserved

Risk displaced from one layer of a structure does not vanish. It reappears elsewhere: in pricing, subordination, contingent obligations, liquidity dependencies, recourse limits, affiliate guarantees, or residual claims. Capital Structures refers to this principle as **Exposure Conservation**.

### Section: Bankruptcy is architectural

Bankruptcy remoteness, non-recourse design, ring-fencing, and true sale are spatial acts as much as legal ones. They establish borders, distance, containment, and controlled adjacency.

### Section: Disclosure is not transparency

Disclosure may be complete, formal, and properly filed without resolving the deeper opacity of the structure. Document sufficiency and intuitive legibility are not the same condition.

### Section: Consolidation changes the picture

Viewed on an unconsolidated basis, a structure may appear clean, discrete, and properly bounded. Viewed on a consolidated basis, the same arrangement may reveal guarantee paths, shared dependencies, contingent support, and live trigger conditions that exceed entity boundaries.

### Section: The diagram is the product

A diagram is not secondary to the structure. It is the structure’s public test. Every arrow, guarantee, exclusion, and trigger path is part of the disclosed arrangement.

### Structure summary boilerplate

This structure arranges ownership, claims, obligations, guarantees, and contingent support across the entities shown below. It may be viewed on an unconsolidated basis, which preserves formal legal boundaries, or on a consolidated basis, which overlays aggregate exposure, dependency, and consequence paths.

### Unconsolidated view description

**Unconsolidated View** respects legal separateness. It presents the structure as constituted by its discrete entities, agreements, and stated recourse limits.

### Consolidated view description

**Consolidated View** overlays the aggregate exposure picture. It shows guarantees, dependencies, residual risks, and consequence paths that persist across formal boundaries.

### Base case description

**Base Case** presents the authorized structure under ordinary operating conditions. Triggers remain unactivated. Guarantees remain contingent. Liquidity remains assumed.

### Trigger state description

**Trigger State** presents the structure after one or more contingent conditions have become active. Consequence paths, collateral demands, guarantee obligations, recourse limitations, or liquidity constraints are shown directly on the diagram.

### Trigger box example

**Trigger Conditions**
- Downgrade below specified rating threshold
- Asset valuation shortfall
- Covenant breach
- Cross-default under related agreement
- Liquidity failure / inability to roll funding
- Margin demand or collateral deficiency
- Parent support event
- Insolvency filing or insolvency-adjacent condition

### Guarantee box example

**Guarantees and Support**
This structure includes specified support arrangements, whether express, implied, reputational, upstream, downstream, affiliate, or contingent. Every guarantee shown is subject to stated conditions, thresholds, or triggering events.

### Residual exposure note example

**Residual Exposure Note**
Although the structure limits recourse in the manner described, residual exposure may persist through affiliate dependency, reputational support expectations, contingent guarantees, shared funding channels, operational reliance, or trigger-linked spillover.

## 9. System for Diagrams

This section should be treated as a formal build specification.

### Diagram principles

Every diagram should obey the following rules:

1. **No arrow exits the frame without a destination.**
2. **Every guarantee has a condition.**
3. **Every isolation mechanism has a residual exposure note.**
4. **Every structure has a stress state.**
5. **Every clean entity map has a consolidated overlay.**
6. **Every trigger has a visible consequence path.**
7. **Every recourse limit is labeled.**
8. **Every support relationship is typed.**
9. **Every boundary has a rationale.**
10. **Every omission is deliberate.**

### What this means in practice

#### Rule 01 — No arrow exits the frame without a destination

If value, guarantee, liquidity, ownership, risk, recourse, servicing, management, or disclosure leaves one entity, the viewer must be able to see where it goes.

#### Rule 02 — Every guarantee has a condition

Guarantees may not float as atmospheric reassurance. Each one should specify the condition under which it becomes active.

Examples:
- active upon payment default
- active upon collateral shortfall
- active upon downgrade
- active upon event of non-performance
- reputational only / not legally binding

#### Rule 03 — Every isolation mechanism has a residual exposure note

Ring-fencing, true sale, bankruptcy remoteness, non-recourse design, and special purpose formation should each carry a note explaining what remains exposed despite the isolation claim.

#### Rule 04 — Every structure has a stress state

A diagram is incomplete if it can only represent the calm state. Every structure must also show what happens when a trigger fires.

#### Rule 05 — Every clean entity map has a consolidated overlay

The unconsolidated map shows formal separation. The consolidated overlay reveals the exposure picture across the whole arrangement.

#### Rule 06 — Every trigger has a visible consequence path

If a rating event, shortfall, default, impairment, or liquidity freeze can occur, the diagram must show what follows: collateral movement, guarantee activation, cash sweep, recourse limit, cross-default, acceleration, or structural failure.

### Required annotation layers

Each diagram should have the ability to show the following layers:

- Entity boundaries
- Ownership paths
- Asset transfer paths
- Cash flow paths
- Guarantee paths
- Trigger paths
- Residual exposure notes
- Recourse labels
- Boundary labels
- Stress-state overlays

### Recommended graphic conventions

- **Solid line:** current legal or economic relationship
- **Dashed line:** contingent or trigger-based relationship
- **Double line:** guarantee or support line
- **Dotted halo / boundary note:** residual exposure or reputational support
- **Overlay color or texture:** trigger state consequences

### Required node metadata on hover / expansion

Each entity node should be able to reveal:
- legal type
- role in structure
- recourse status
- support obligations
- dependencies
- trigger relevance
- related filings

### Required edge metadata on hover / expansion

Each arrow / edge should be able to reveal:
- relationship type
- source
- destination
- active or contingent status
- condition precedent
- effect if triggered
- residual note if applicable

## 11. Glossary Additions

The glossary should expand to include the following entries.

### Exposure Conservation
**Definition:** The proposition that risk displaced from one layer of a structure must reappear elsewhere rather than disappear.

**Capital Structures usage:** A doctrine of structural thermodynamics.

### Non-Recourse
**Definition:** A financing arrangement in which recovery is limited to specified assets and does not extend to the originator’s remaining estate.

**Capital Structures usage:** A boundary drawn in advance for where loss is permitted to stop.

### Bankruptcy Remoteness
**Definition:** A legal and structural condition designed to reduce the risk that an entity’s assets will be pulled into an affiliate’s insolvency.

**Capital Structures usage:** Distance engineered against contagion.

### True Sale
**Definition:** A transfer characterized as an actual sale rather than secured borrowing, typically to preserve asset separation in insolvency.

**Capital Structures usage:** A legal transformation on which the rest of the structure depends.

### Consolidated View
**Definition:** A perspective that evaluates the structure on an aggregate basis, overlaying exposure, guarantees, dependencies, and consequence paths across entity boundaries.

**Capital Structures usage:** The view in which containment is tested.

### Unconsolidated View
**Definition:** A perspective that respects formal entity boundaries and presents the structure according to its discrete legal components.

**Capital Structures usage:** The version the structure is authorized to present of itself.

### Trigger State
**Definition:** The condition of a structure after a contractual, rating, collateral, liquidity, or governance event has activated contingent consequences.

**Capital Structures usage:** The moment the diagram begins to move.

### Residual Exposure
**Definition:** Risk that remains after transfer, isolation, ring-fencing, or non-recourse drafting, whether by contingency, support expectation, operational dependency, or system linkage.

**Capital Structures usage:** What the structure cannot fully exile.

### Liquidity Event
**Definition:** A transaction or financing event that converts constrained claims into available cash for continuation, repayment, or survival.

**Capital Structures usage:** Cash introduced where time has started to harden.

### Aggregate Dependency
**Definition:** A functional reliance that persists across formal legal boundaries, including shared funding, servicing, management, guarantees, or operational support.

**Capital Structures usage:** Separation that continues to depend.

```ts
interface StructurePageData {
  id: string
  title: string
  status: 'authorized' | 'pending' | 'effective' | 'amended' | 'suspended' | 'triggered' | 'dissolved'
  date: string
  summary: string
  entities: EntityNode[]
  edges: StructureEdge[]

  id: string
  name: string

  recourseStatus?: string
  dependencies?: string[]
  notes?: string[]
}

interface StructureEdge {
  id: string
  source: string
  destination: string
  type: 'ownership' | 'asset-transfer' | 'cash-flow' | 'guarantee' | 'servicing' | 'management' | 'trigger-path' | 'support'
  activeInBaseCase: boolean
  activeInTriggerState: boolean
  condition?: string
  recourseLabel?: string
  residualNote?: string
}

interface TriggerCondition {
  id: string
  label: string
  description: string
  consequencePathIds: string[]
}
```

### Behavior notes

1. The diagram should redraw or visibly change when toggles change.
2. Consolidated mode should introduce overlays rather than replacing the entire entity map.
3. Trigger state should visibly activate paths, support lines, or notes that are dormant in base case.
4. Residual exposure notes should be inspectable on nodes and edges.
5. Guarantee lines should never appear without a visible condition.
6. The viewer should always be able to identify what has changed when switching modes.

### UI principle

The interaction should make the viewer feel that the structure is stable, then reveal that stability as contingent, conditional, and perspective-dependent.
